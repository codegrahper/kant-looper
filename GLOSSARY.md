# GLOSSARY

칸트루퍼 개발·운영 중에 쓰인 용어 사전. 개발자가 아니어도 클로드와 소통할 수
있도록, 코드에는 등장하지만 대화에 직접 드러나지 않는 용어까지 포함한다.
클로드가 호출한 서브에이전트 안에서만 쓰인 용어도 포함한다. 새 용어가 나올
때마다 이 브랜치에 계속 누적한다.

---

## Git 개념

**SSOT** : (Single Source of Truth)는 데이터나 정보의 신뢰할 수 있는 기준을
하나의 출처로 통일하여 관리하는 원칙.
> 라우팅 가이드를 참고 자료로 삼되, 실제로 무엇을 선택할지는 매번 클로드가
> 판단한다 — 문서가 유일한 근거지만 그걸 기계가 강제로 파싱해서 따르진 않는다.

**ff-only** : (Fast-Forward Only)는 브랜치 병합 시 병합 커밋을 만들지 않고,
브랜치 포인터를 앞으로 이동하는 방식만 허용하는 옵션.
> 개발 브랜치에서 만든 커밋을 배포판(`main`)에 반영할 때 `git merge --ff-only`만
> 쓴다 — 병합 커밋이 안 생기니 히스토리가 한 줄로 깔끔하게 유지된다.

**브랜치(branch)** : 하나의 저장소 안에서 원본에 영향을 주지 않고 독립적으로
작업할 수 있게 나눠놓은 작업 줄기.
> 외부 도구가 작업할 때마다 `agent/kant/<작업이름>` 같은 새 브랜치를 만들어
> 그 안에서만 커밋한다 — main 브랜치는 항상 깨끗하게 유지된다.

**worktree** : 하나의 git 저장소를 여러 개의 독립된 작업 폴더로 동시에 열어두는
기능. 브랜치가 달라도 같은 `.git` 히스토리를 공유하면서, 실제 파일은 서로
다른 폴더에 격리된다.
> 외부 도구를 부를 때마다 `/tmp/kant-worktree-<번호>` 같은 임시 폴더를
> 새로 만들어 그 안에서만 작업시킨다 — 원본 저장소에 영향을 주지 않는다.

**diff** : 변경 전과 변경 후를 비교해서 무엇이 달라졌는지만 뽑아 보여주는 것.
> 에이전트가 "다 고쳤다"고 말해도 그 말만 믿지 않고, 실제로 git diff에
> 변경 내용이 있는지 항상 대조해서 확인한다.

**merge commit** : 두 브랜치를 합칠 때 그 합침 자체를 기록하는 별도의 커밋.
칸트루퍼는 이걸 금지하고 ff-only만 허용한다.
> `safety-promises.md`에 "merge commit(no-ff) 금지"가 절대 규칙으로 박혀 있다.

**HEAD** : 지금 내가 보고 있는(체크아웃한) 커밋을 가리키는 포인터. "지금 여기"
표시라고 생각하면 된다.
> 커밋을 만든 직후 `git rev-parse HEAD`로 방금 만든 커밋의 해시를 얻어 기록한다.

**detached HEAD** : 브랜치가 아니라 특정 커밋 하나에 HEAD가 붙어있는 상태.
브랜치 이름 없이 "여기"만 가리키고 있어서, 그 상태로 커밋하면 나중에 찾기 어려워진다.
> 현재 브랜치 이름이 `HEAD`(=detached 상태)로 나오면 작업 브랜치 생성을
> 거부하도록 안전장치를 걸어뒀다.

**rebase** : 커밋들을 다른 기준점 위로 다시 쌓아 올려 히스토리를 재작성하는 명령.
> 브랜치를 망가뜨릴 위험이 커서 칸트루퍼는 이 명령을 절대 자동 실행하지 않는다
> (스크립트 자체 점검에서 이 명령이 있으면 실패 처리).

**reset --hard** : 현재 브랜치를 특정 지점으로 되돌리면서 그 이후 변경사항을
전부 버리는 명령.
> rebase와 마찬가지로 되돌릴 수 없는 손실을 일으킬 수 있어 금지 목록에 있다.

**branch -D (강제 삭제)** : 병합 여부와 상관없이 브랜치를 강제로 지우는 명령.
> 작업 중이던 브랜치가 실수로 사라질 수 있어 자동 실행 금지 목록에 있다.

**git tag** : 특정 커밋에 이름표(보통 버전 번호)를 붙여 나중에 쉽게 찾을 수 있게
하는 것.
> 버전이 나올 때마다 `git tag v0.X.Y`로 그 버전이 정확히 어느 커밋인지
> 표시해둔다.

**staged / unstaged / untracked** : git이 파일 변경을 세 단계로 구분하는 방식 —
staged(커밋 준비 완료), unstaged(수정했지만 아직 준비 안 함), untracked(한 번도
git에 등록된 적 없는 새 파일).
> 안전 검사가 이 세 가지를 전부 따로 확인한다 — 어느 하나만 봐서는 실제로
> 뭐가 바뀌었는지 놓칠 수 있어서다.

**stash** : 지금 하던 작업을 커밋하지 않고 잠시 서랍에 넣어뒀다가 나중에 다시
꺼내 쓸 수 있게 하는 기능.
> 워크트리가 깨끗한지 검사할 때, 서랍(stash)에 숨겨둔 변경사항까지 있으면
> "깨끗하지 않다"고 판단해서 작업을 거부한다.

**tree 해시 / tree mismatch** : git이 "이 커밋 시점의 전체 파일 구조"를 나타내는
고유 해시값. 검토한 시점의 tree와 실제로 커밋된 tree가 다르면 뭔가 중간에
바뀐 것이다.
> 리뷰가 끝난 뒤 실제로 커밋된 내용이 리뷰 당시와 정확히 같은지 이 해시를
> 비교해서 확인한다 — 다르면 즉시 실패 처리한다.

**remote / origin** : 내 컴퓨터 밖, 인터넷 어딘가(예: GitHub)에 있는 저장소
사본. `origin`은 그 원격 저장소를 가리키는 기본 이름이다.
> 어떤 원격에도 자동으로 push하지 않는다 — 사람이 직접 push를 실행해야 한다.

**git hooks** : 커밋·푸시 같은 git 작업 전후에 자동으로 실행되게 걸어둔 스크립트.
> 커밋할 때 로컬에 설정된 다른 hook이 끼어들지 않도록, 빈 hook 경로를
> 임시로 지정해서 커밋한다.

**checkout -B** : 지정한 이름의 브랜치를 새로 만들면서(이미 있으면 덮어쓰며)
그쪽으로 이동하는 명령.
> 새 worktree를 만들 때마다 `git checkout -B <새브랜치>`로 격리된 작업
> 브랜치를 함께 만든다.

## 프로세스·시스템

**PID** : (Process ID) 컴퓨터에서 지금 실행 중인 프로그램 하나하나에 매겨지는
고유 번호.
> 임시 작업 폴더 이름에 PID를 넣어서(`/tmp/kant-worktree-12345`), 동시에 여러
> 작업이 겹쳐도 폴더 이름이 충돌하지 않게 한다.

**nohup** : 터미널 창을 닫아도(또는 이 명령을 실행한 프로그램이 끝나도) 그
안에서 실행 중인 명령이 계속 살아있게 해주는 명령.
> `--detach`로 백그라운드 실행할 때 `nohup`으로 감싸서, 원래 요청이 끝나도
> 실제 작업은 끊기지 않고 계속되게 한다.

**detach / 백그라운드 실행** : 명령을 실행시켜 놓고 그 결과를 기다리지 않고
바로 다음 일을 할 수 있게 하는 방식. 반대말은 포그라운드(그 자리에서 끝날
때까지 기다리는 방식).
> 짧은 작업은 그냥 기다려도(foreground) 되고, 오래 걸릴 작업만 detach를 쓴다.

**exit code (종료 코드)** : 프로그램이 끝날 때 "성공했는지 실패했는지, 어떻게
실패했는지"를 숫자로 알려주는 값. 0은 보통 "정상 종료"를 뜻한다.
> 0=정상, 124=시간초과, 200=인프라 오류처럼 숫자마다 의미를 미리 정해두고,
> 그 숫자로 다음에 뭘 할지(재시도할지, 포기할지) 자동으로 판단한다.

**signal (SIGTERM/SIGKILL)** : 운영체제가 실행 중인 프로그램에게 "이제 그만
멈춰라"라고 보내는 신호. SIGTERM은 "정중하게 끝내달라", SIGKILL은 "무조건
즉시 강제 종료"에 가깝다.
> 작업이 제한 시간을 넘기면 먼저 SIGTERM으로 부드럽게 종료를 요청하고, 그래도
> 안 죽으면 5초 뒤 SIGKILL로 강제 종료한다.

**process group** : 서로 연관된 여러 프로세스를 하나의 묶음으로 다루는 단위.
> 외부 도구를 부를 때 별도 process group으로 띄워서, 시간초과로 죽일 때
> 관련된 프로세스 전체를 한 번에 정리할 수 있게 한다.

**stdin / stdout / stderr** : 프로그램이 입력을 받는 통로(stdin), 정상 출력을
내보내는 통로(stdout), 오류·로그를 내보내는 통로(stderr) 세 가지 기본 통로.
> 외부 CLI를 부를 때 stdin은 막아서 대화형으로 멈추지 않게 하고, stdout은
> 응답 파일로, stderr는 로그 파일로 각각 따로 저장한다.

**환경변수(environment variable)** : 프로그램 실행 전에 미리 설정해두는 값으로,
코드를 안 고치고도 동작을 바꿀 수 있게 해준다.
> `KANT_AUTO_COMMIT`, `KANT_STATE_ROOT` 같은 환경변수로 커밋 여부, 상태 저장
> 위치 같은 걸 코드 수정 없이 바꿀 수 있다.

**shebang** : 스크립트 파일 맨 첫 줄에 있는 `#!`로 시작하는 줄. "이 파일을
어떤 프로그램으로 실행해야 하는지" 운영체제에 알려준다.
> 모든 셸 스크립트 첫 줄이 `#!/usr/bin/env bash`로 시작한다.

**timeout** : 정해진 시간 안에 끝나지 않으면 강제로 중단시키는 장치.
> 외부 도구 호출마다 역할별로 다른 제한 시간(구현은 30분, 리뷰는 15분 등)을
> 걸어서, 하나가 멈춰버려도 전체가 무한정 기다리지 않게 한다.

**subprocess** : 어떤 프로그램이 실행 도중에 새로 띄우는 또 다른 프로그램.
> 시간제한을 정확히 걸기 위해 파이썬으로 외부 CLI를 subprocess로 띄우고
> 감시한다.

**cwd (현재 작업 디렉터리)** : 명령이 "지금 어느 폴더를 기준으로" 실행되는지를
가리키는 값.
> 이 값이 확실치 않으면 실수로 엉뚱한 폴더에서 실행될 위험이 있어서,
> 반드시 명시적으로 지정하게 하고 없으면 조용히 넘어가지 않고 바로 실패시킨다.

**symlink (심볼릭 링크) / realpath** : symlink는 실제 파일이 아니라 다른
파일·폴더를 가리키는 "바로가기". realpath는 그 바로가기를 다 따라가서 진짜
실제 경로를 알아내는 것.
> 경로가 symlink를 통해 저장소 밖으로 몰래 빠져나가는 걸 막기 위해, 항상
> realpath로 정규화한 뒤 안전 여부를 판단한다.

## 데이터 형식·프로토콜

**JSON** : 텍스트로 데이터를 구조화해서 표현하는 대표적인 형식. `{"키": "값"}`
처럼 생겼다.
> 에이전트의 판정(verdict)은 자유 텍스트가 아니라 정해진 JSON 형식으로
> 나와야 한다 — 그래야 기계가 정확히 파싱할 수 있다.

**JSON Lines (JSONL)** : 한 줄에 JSON 객체 하나씩, 여러 줄로 이어지는 형식.
전체를 한 번에 파싱하는 대신 한 줄씩 순서대로 읽을 수 있다.
> 일부 CLI 도구의 실행 로그가 JSONL 형태라, 마지막 줄(최종 이벤트)만 골라
> 읽는 식으로 처리한다.

**JSON Schema** : "이 JSON은 이런 모양이어야 한다"를 정해두는 명세.
> 도구에 따라 이 스키마 파일을 함께 넘겨서, 응답이 정해진 필드·형식을
> 벗어나지 않게 유도한다.

**MCP** : (Model Context Protocol)은 AI 모델이 외부 도구·데이터에 정해진
방식으로 접근할 수 있게 하는 공통 규격.
> agy는 Stitch(Google의 UI 디자인 생성 도구)에 MCP로 이미 연결돼 있지만,
> 프롬프트에서 먼저 호출하라고 시키지 않으면 그냥 넘어간다.

**정규식(regex)** : 문자열 안에서 특정 패턴을 찾아내는 규칙 표현 방식.
> 비밀번호나 API 키처럼 생긴 문자열 패턴을 정규식으로 미리 정의해두고,
> 로그나 코드에 그런 패턴이 있으면 자동으로 걸러낸다.

**envelope (엔벨로프) / 언랩(unwrap)** : 실제로 필요한 데이터가 다른 바깥
구조 안에 한 겹 더 싸여서 오는 것(envelope), 그 바깥 껍데기를 벗겨 안의
진짜 데이터를 꺼내는 것(unwrap).
> 어떤 도구는 판정 결과를 `{"result": {...진짜 내용...}}`처럼 한 겹 싸서
> 주는데, 그 겉포장을 벗기고 안쪽만 꺼내 쓴다.

**코드펜스(code fence)** : 마크다운에서 코드 블록을 \`\`\`로 감싸는 표기.
> 모델이 JSON을 답할 때 종종 \`\`\`json ... \`\`\` 처럼 코드블록으로 감싸서
> 주는데, 그 껍데기를 벗기고 안의 JSON만 추출해서 쓴다.

## 테스트·품질

**mock** : 진짜 대신 가짜로 흉내만 낸 것. 실제 도구를 안 부르고도 그 도구를
부른 것처럼 테스트할 수 있게 해준다.
> 실제로 값비싼 외부 CLI를 매번 부르지 않고, mock(가짜) 버전으로 빠르게
> 회귀 테스트를 돌린다.

**assertion (어서션)** : 테스트 코드 안에서 "이게 이 값과 같아야 한다"처럼
확인하는 검사 한 줄 한 줄.
> 테스트를 돌리면 "32개 assertion 통과"처럼 몇 개의 검사를 통과했는지
> 세어서 보여준다.

**회귀 테스트(regression test)** : 코드를 고친 뒤에도 예전에 되던 게 여전히
되는지 확인하는 테스트.
> 뭘 하나 고칠 때마다 전체 회귀 테스트를 돌려서, 그 수정이 다른 걸 망가뜨리지
> 않았는지 확인한다.

**E2E 테스트(End-to-End)** : 부분만 따로 테스트하지 않고, 실제 사용자가 쓰는
것과 똑같은 전체 흐름을 처음부터 끝까지 돌려보는 테스트.
> 실제 git worktree를 만들어서 진짜 커밋까지 이어지는 전체 과정을 통째로
> 검증하는 e2e 테스트가 별도로 있다.

**정적 검사(static analysis)** : 코드를 실제로 실행하지 않고, 코드 자체만
읽어서 문제가 있는지 확인하는 검사.
> 일부 테스트는 코드를 실행하지 않고 정적 검사만 하고, 실제 동작 확인은
> 별도의 시나리오 테스트가 담당한다.

**dry-run** : 실제로 실행하지는 않고, "실행하면 어떻게 될지"만 미리 보여주는
모드.
> `--dry-run`으로 실행하면 실제로 아무것도 바꾸지 않고 환경 점검·계획만
> 보여준다.

**fixture (픽스처)** : 테스트를 위해 미리 준비해둔 샘플 데이터나 상황.
> 다양한 상황을 가정한 문서형 fixture를 여러 개 만들어두고, 그걸로
> 라우팅 판단이 맞는지 검증한다.

**self-test (자체 점검)** : 스크립트가 실행되기 전에 스스로 자기 자신에게
문제가 없는지 검사하는 것.
> 안전 검사 스크립트 자체에 금지된 git 명령(rebase, reset --hard 등)이
> 숨어 있지 않은지, 실행 전에 스스로 점검한다.

**idempotency (멱등성) / 재진입 가드** : 같은 조작을 여러 번 반복해도 결과가
한 번 한 것과 똑같이 유지되는 성질. 재진입 가드는 "이미 한 번 처리한 건
또 처리하지 않게" 막는 장치.
> 같은 수정 제안이 실수로 두 번 들어와도 중복 적용되지 않도록 표식을
> 남겨서 막는다.

## LLM·에이전트 개념

**verdict** : 외부 도구(에이전트)가 작업을 마친 뒤 내리는 최종 판정. "완료했다"는
말이 아니라 `PASS`(통과) / `CHANGES_REQUESTED`(수정 필요) / `BLOCKED`(막힘) /
`INVALID_OUTPUT`(판정 자체를 못 냄) 중 하나로 정해진 형식(JSON)으로 나와야 한다.
> codex가 포트폴리오 사이트 리뷰에서 `CHANGES_REQUESTED`를 냈을 때, 그 안에
> "왜 안 되는지"까지 정확히 담겨 있어서 실제로 유효한 지적이었다.

**어댑터(adapter)** : 서로 다른 인터페이스를 가진 것들을 같은 방식으로 다룰 수
있게 중간에서 변환해주는 프로그램.
> codex/grok/opencode/agy/claude 다섯 개 CLI 도구는 실행 방식이 전부 다른데,
> 각 도구 전용 어댑터 스크립트가 "프롬프트 받아서 verdict 돌려주기"라는
> 같은 규격으로 맞춰준다.

**디스패처(dispatcher)** : 들어온 작업이나 이벤트를 보고 어디로 보낼지 결정해서
넘겨주는 역할을 하는 프로그램.
> 에이전트끼리 결과를 자동으로 다음 에이전트에 넘기는 디스패처를 만들어봤지만,
> 판정을 스크립트에 맡기는 방식이 계속 문제를 일으켜 걷어내고 클로드가
> 직접 다음 작업을 판단하는 방식으로 되돌렸다.

**콜백(callback)** : 어떤 작업이 끝났을 때 "끝났다"고 자동으로 알려주는 신호나
그 신호를 받아서 실행되는 처리.
> 백그라운드로 실행한 작업이 끝나면 하네스가 알아서 완료 알림을 보내주는데,
> 이게 콜백이다. 걸어두지 않으면 실제로 끝났는지 아무도 모르는 상태가 된다.

**게이트(gate)** : 다음 단계로 넘어가기 전에 반드시 통과해야 하는 자동 검사
관문.
> 외부 도구가 코드를 고친 뒤, 실제로 변경사항이 있는지·안전 규칙을 어기지
> 않았는지 같은 게이트를 전부 통과해야만 커밋까지 이어진다.

**샌드박스(sandbox)** : 실제 환경에 영향을 주지 못하게 격리된, 제한된 실행
공간.
> agy를 부를 때 "읽기 전용"과 "쓰기 허용" 두 가지 샌드박스 모드가 있다.
> 리뷰할 때는 읽기 전용으로 묶어서, 실수로 파일을 고치는 사고를 막는다.

**HPRAR** : Plan(계획)→Implement(구현)→Review(검토)→Repair(수정)를 사람 개입
없이 자동으로 반복시키려던, 지금은 포기한 실행 방식의 이름.
> "자동으로 계속 도는 루프"를 여러 번 시도했지만 매번 같은 지점(판정을
> 기계에 맡기는 부분)에서 무너져서 포기했다.

**토큰(token)** : AI 모델이 글을 읽고 쓸 때 계산하는 최소 단위이자, 그만큼
쓴 양에 따라 비용·사용량이 매겨지는 기준.
> 클로드의 사용량은 제한적이라, 직접 코드를 다 짜기보다 다른 도구에
> 위임해서 클로드는 판단·검증에만 토큰을 쓰는 게 칸트루퍼가 존재하는 이유다.

**프롬프트(prompt)** : AI 모델에게 무엇을 하라고 전달하는 입력 텍스트.
> 프롬프트 원문은 그대로 로그에 남기지 않고 "[내용 생략: N자]"처럼
> 가려서 기록한다 — 민감한 내용이 로그에 그대로 남지 않게 하기 위해서다.

**reasoning effort (추론 강도)** : 모델이 답을 내기 전에 얼마나 깊게 "생각"하는
데 자원을 쓸지 정하는 설정. 높을수록 신중하지만 느리고 비싸진다.
> 복잡한 작업은 높은 reasoning effort, 단순 반복 작업은 낮은 effort로
> 나눠서 부른다.

**컨텍스트(context) / 컨텍스트 윈도우** : 모델이 한 번에 "기억하고 참고할 수
있는" 텍스트의 최대 범위.
> "1M 컨텍스트" 모델은 아주 큰 저장소나 긴 대화를 통째로 참고할 수 있어서,
> 대형 리팩터링 같은 장기 작업에 쓴다.

**모델군(model tier/family)** : 하나의 브랜드 이름 아래 성능·비용이 다른 여러
모델이 묶여있는 것.
> "Codex 5.6"은 모델 하나가 아니라 상위/균형/효율형 세 모델이 묶인
> 모델군이다 — 작업 난도에 따라 그 안에서 골라 쓴다.

**공급자(provider)** : 그 모델을 실제로 서비스하는 회사/플랫폼.
> 같은 도구 안에서도 어떤 모델을 부르느냐에 따라 요금이 청구되는 공급자
> (예: Z.AI Coding Plan, MiniMax Token Plan)가 서로 다르다.

**모델 레지스트리(model registry)** : 지금 실제로 선택 가능한 모델들의 공식
목록.
> 어떤 모델을 후보로 둘지는 사람이 정하고 유지하며, 그 범위 안에서 클로드가
> 작업마다 판단해서 고른다.

**하네스(harness)** : 에이전트를 실제로 실행하고 도구 호출·결과를 오가게
해주는 실행 환경/틀.
> Antigravity(agy)는 모델 이름이 아니라 Google의 에이전트 실행 하네스다 —
> 그 안에서 Gemini 모델을 골라 쓴다.

**권한 모드(permission-mode)** : 도구가 파일 편집이나 명령 실행을 자동으로
승인할지, 매번 물어볼지를 정하는 설정.
> 리뷰할 때는 자동 승인을 꺼서 실수로 파일이 고쳐지지 않게 하고, 구현할
> 때만 자동 승인을 켠다.

**재시도 루프(retry loop)** : 실패하면 같은 시도를 계속 반복하는 것. 잘못
설계하면 똑같은 실수를 무한히 반복하는 함정이 된다.
> 모델에게 "도구를 쓴 직후에도 반드시 정해진 형식으로 답하라"고 명시해서
> 이런 무한 반복 함정에 빠지지 않게 한다.

**에스컬레이션(escalation) / 상향** : 낮은 단계에서 해결이 안 되면 더 강한
수단(더 비싼 모델, 사람 등)으로 넘기는 것.
> 두 가지를 구분해야 한다. (1) **작업 난이도에 따른 자동 모델 승격**(쉬운 작업은
> 싼 모델, 어려우면 자동으로 비싼 모델로)은 쓰지 않는다 — 어떤 도구·모델로 시작할지는
> 클로드가 그때그때 판단한다. (2) 그러나 **실행 실패에 대한 자동 failover**는 존재한다 —
> 고른 모델이 실패하면 `fallback-dispatcher.sh`가 같은 난이도 티어(T0~T3)의 다른
> 모델부터 자동으로 시도하고, 최종적으로 claude까지 이어진다. 즉 "난이도 기반 자동
> 승격"은 없지만 "실패 기반 자동 대체"는 있다.

## CLI 관례

**flag / 옵션 / 서브커맨드** : flag는 `--detach`처럼 명령 뒤에 붙는 설정 스위치,
서브커맨드는 `run`, `status`처럼 그 프로그램 안의 "하위 명령"을 뜻한다.
> `kant-loop.sh run TASK.md --quick --detach`에서 `run`이 서브커맨드,
> `--quick`/`--detach`가 플래그다.

**positional argument (위치 인자)** : 이름 없이 그냥 순서(위치)로 의미가
정해지는 인자.
> agy CLI는 프롬프트를 순서상 맨 뒤에 그냥 끼워 넣으면(위치 인자로 주면)
> 인식하지 못하고 무시해버리는 버그가 있었다 — 반드시 `-p 값` 형태로
> 넘겨야 했다.

**--dry-run** : 실제로 실행하지 않고, 실행하면 어떻게 될지만 미리 확인하는
플래그.

**--chain** : `도구:모델,도구:모델,...` 형식으로 구현→검토→수정을 순서대로
이어 붙이는 지정 방식.

**--existing-worktree** : 새 작업 폴더를 만들지 않고, 이미 만들어둔 worktree를
그대로 재사용하겠다는 플래그.
> 같은 worktree 안에서 후속 작업(예: 이전 구현을 이어서 리뷰)을 시킬 때
> 이 옵션으로 새 폴더 생성을 건너뛴다.

**--no-auto-commit** : 검증까지 통과해도 자동으로 커밋하지 않고 사람 결정을
기다리게 하는 플래그.

## 칸트루퍼 고유 용어

**quick 모드** : 도구 하나(또는 구현→검토→수정 3단계)를 한 번 부르고 끝내는
기본 실행 방식. 자동으로 계속 도는 방식(HPRAR)의 반대 개념.

**pass_no_commit** : 판정은 통과(PASS)했지만 아직 커밋은 하지 않은 상태를
나타내는 결과값 — `--no-auto-commit`이거나 사람 확인을 기다릴 때 이 상태가 된다.

**role (역할: implement / review / repair)** : 지금 이 호출이 "무엇을
하는 차례인지"를 나타내는 값. role에 따라 프롬프트와 권한(읽기 전용/쓰기 허용)이
달라진다. 실제 실행(quick / quick-chain / parallel)에서 쓰이는 role은
`implement`·`review`·`repair` 셋뿐이다 — 옛 `plan`·`verify`는 어댑터의 sandbox
분기에 케이스만 남은 잔재이고 kant-loop.sh가 그 role로 호출하지는 않는다.

**repo_hash** : 지금 작업 중인 저장소 경로를 해시로 바꿔서, 저장소별로 상태
저장 폴더를 서로 안 섞이게 나누는 내부 식별자.

**run_id** : 한 번의 실행을 구별하는 고유 이름. 상태 조회·보고서 생성 등을
할 때 이 아이디로 그 실행을 다시 찾는다.

**state_dir** : 한 번의 실행에 대한 로그·결과·설정을 모아두는 전용 폴더.

**phase-events.log** : 한 실행 안에서 일어난 일들(호출 시작, 판정, 커밋 등)을
시간 순서대로 기록한 로그 파일.

**무진전 감지(no-progress detection)** : 같은 수정을 반복하거나, 같은 테스트가
계속 실패하는 등 "더 나아지지 않는" 상태를 감지해서 자동으로 멈추던 장치.
> **v0.8에서 제거됨** — 이 감지기(`no-progress-detector.sh`)는 실제로 호출하는
> 곳이 전혀 없는 죽은 코드였다. 지금은 무한 반복을 role별 제한 시간(timeout)과
> fallback 체인 소진으로만 막는다. 별도의 "같은 diff/테스트 반복" 감지는 없다.

**Nomad 원칙 / 스위처빌리티(switchability)** : 특정 회사·모델에 정착하지 않고,
새 모델·공급사가 나오면 언제든 자유롭게 옮겨갈 수 있게 설계하는 원칙.

**다극화(multi-polar)** : 특정 빅테크 하나에 의존하지 않고, 여러 공급사를
동등하게 두고 평가하는 방향.

**판단·위임 원칙** : "어떤 작업에 어떤 도구를 쓸지" 같은 판단을 고정된 스크립트
로직이 대신하지 않고, 그때그때 클로드가 직접 하게 하는 설계 원칙.

**Stitch** : Google의 AI UI 디자인 생성 도구. agy(Antigravity)에 MCP로 이미
연결돼 있지만, 프롬프트에서 명시적으로 부르라고 하지 않으면 그냥 지나간다.

**Hermes / H1·H2 패턴** : 다른 에이전트(Henry)가 쓰는 완료 알림 방식의 이름.
전체 실행을 백그라운드+완료알림으로 감싸는 방식(H1), `--detach` 후 완료 대기를
따로 백그라운드+완료알림으로 감싸는 방식(H2) 두 가지가 있다. 클로드는 같은
도구는 못 쓰지만, `run_in_background` 옵션으로 같은 효과를 낼 수 있다는 걸
확인했다.
